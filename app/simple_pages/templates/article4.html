SOLID

  • The Single Responsibility Principle
  • The Open-Closed Principle
  • The Liskov Substitution Principle
  • The Interface Segregation Principle
  • The Dependency Inversion Principle

    • Single Responsibility Principle

    A class should have one, and only one, reason to change.

    One class should serve only one purpose. This does not imply that each class should have only one method, but they should all relate directly to the responsibility of the class. All the methods and properties should work towards the same goal. When a class serves multiple purposes or responsibilities, it should be made into a new class.

    Open-Closed Principle

    Entities should be open for extension, but closed for modification.

    This principle describes how code should be written in a way that allows for the developer to add new functionality without changing the existing code. This would prevent a situation where a change to one of the classes also requires the adaptation of all dependent classes.

    Liskov Substitution Principle

    Let φ(x) be a property provable about objects x of type T. Then φ(y) should be true for objects y of type S where S is a subtype of T.

    This principle states that any implementation of an abstraction (interface) should be substitutable in any place that the abstraction is accepted. When applying this principle, the behavior of the classes becomes more important than the structure.

    Interface Segregation Principle

    A client should not be forced to implement an interface that it doesn't use.

    The goal of this principle is to minimize side consequences and repetition by dividing the software into multiple, independent parts. This principle is easy to violate, unfortunately, because when software evolves, more features are added as required/requested.

    Dependency Inversion Principle

    High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details. Details should depend on abstractions.

    This principle explains how both high-level and low-level modules depend on the abstraction. There are two dependencies: the high-level module which depends on the abstraction and the low-level which depends on the same abstraction. An abstraction means to create an interface or an abstract class which is not concrete.
